# チケット厳格査読フロー（思考OS準拠）

## 概要
作成されたチケットが「推測禁止」「根拠に基づく設計」を満たしているか、実装前にAIが自ら厳しく批判・検証するフロー。

## 依頼フレーズ
- `チケットを厳しく査読して`
- `このチケットに不備はないか？`

## 査読観点（AIへのチェックリスト）

### 1. 正常系の欠如チェック
- **最優先**: 成功時の挙動（Happy Path）が定義されているか？
- **詳細**: ステータスコード 200 時のレスポンス形式、画面への反映方法が具体的か。

### 2. インターフェースの整合性
- **データ構造**: リクエスト/レスポンスの JSON フィールド名が定義されているか？
- **制約**: 既存の `internal/` 配下の構造体と矛盾していないか。

### 3. 疎結合性の検証
- **クライアント**: UIライブラリや特定の環境に依存した記述がないか？
- **バックエンド**: リポジトリパターンが抽象化（interface）として定義される前提になっているか。

### 4. 数値とエビデンス
- **パフォーマンス**: タイムアウト秒数などの数値が明記されているか。
- **Done定義**: カバレッジ率などの客観的指標が含まれているか。

## 実行アクション
1. 不備が見つかった場合、実装には進まず、ユーザーに「修正案」を提示し、チケットの再生成を提案せよ。
2. 全て合格した場合のみ、「実装準備完了」と宣言せよ。

# BDD完了時の一括更新フロー

## 概要
1つの振る舞い（BDD）の実装が完了し、テストがすべてパスした際に実行する標準フローです。
「生きた仕様書」の鮮度を保ち、GitHub Copilotが次回の実装で既存機能を再利用できる状態を整えます。

## 依頼フレーズ
- `BDD完了したから締めて`
- `一括更新フロー実行`

## 実行プロセス（AIへの指示）

### 1. テスト実行とエビデンス確保
- `scripts/run-tests.sh` を実行せよ。
- **合格条件**: ステータスが `PASS` であること。失敗（FAIL）した場合は、修正案を提示し、以降のステップを中断せよ。
- **記録**: 出力された「Total Coverage」の数値を一時保持せよ。

### 2. 構造・シンボル情報の最新化
- `scripts/reload-callvis.sh` を実行せよ。
- **確認**: `docs/symbols_map.md` と `docs/symbols/*.md` が更新されたことを確認せよ。これにより、次のBDDでCopilotが新設関数を「既存資産」として認識可能になる。

### 3. ADR（意思決定記録）の自動更新
- `cat-cmd/adr-gen.md` のアルゴリズムに従い、既存ファイルへの「追記」か「新規作成」かを選択せよ。
- **必須項目**: 
    - 今回の実装で再利用した既存のドメイン/関数。
    - 却下した代替案とその理由。
    - **Test Evidence**: ステップ1で取得した「カバレッジ率」を必ず記載せよ。

### 4. インデックスとチケットの同期
- `scripts/update-adr-index.sh` を実行し、ADR目次を最新化せよ。
- `tickets/active/` 内の該当するチケットファイルを `tickets/done/` へ移動（`mv`）せよ。

## 確認事項（AIへの最終チェック）
1. [ ] **テスト数値**: ADRの末尾にカバレッジ率（例: 85.2%）が明記されているか？
2. [ ] **シンボル地図**: `docs/symbols_map.md` に今回の新設パッケージ/関数が含まれているか？
3. [ ] **チケット移動**: Obsidianの `active` フォルダからファイルが消え、`done` に移動したか？
4. [ ] **次への布石**: `docs/adr/README.md` が更新され、次のCopilot作業で参照可能になっているか？

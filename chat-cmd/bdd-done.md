# BDD完了時の一括更新フロー

## 概要
1つの振る舞い（BDD）の実装が完了し、テストがすべてパスした際に実行する標準フローです。
「生きた仕様書」の鮮度を保ち、GitHub Copilotが次回の実装で既存機能を再利用できる状態を整えます。

## 依頼フレーズ
- `BDD完了したから締めて`
- `一括更新フロー実行`

## 実行プロセス（AIへの指示）

### 1. テスト実行とエビデンス確保
- `scripts/run-tests.sh` を実行せよ。
- **合格条件**: ステータスが `PASS` であること。失敗（FAIL）した場合は、修正案を提示し、以降のステップを中断せよ。
- **記録**: 出力された「Total Coverage」の数値を一時保持せよ。

### 2. 構造・シンボル情報の最新化
- `scripts/reload-callvis.sh` を実行せよ。
- **確認**: `docs/current_symbols.md` が更新されたことを確認せよ。これにより、次のBDDでCopilotが新設関数を「既存資産」として認識可能になる。

### 3. ADR（意思決定記録）の自動更新
- `cat-cmd/adr-gen.md` のアルゴリズムに従い、既存ファイルへの「追記」か「新規作成」かを選択せよ。
- **必須項目**: 
    - 今回の実装で再利用した既存のドメイン/関数。
    - 却下した代替案とその理由。
    - **Test Evidence**: ステップ1で取得した「カバレッジ率」を必ず記載せよ。

### 4. インデックスとチケットの同期（階層管理ルール）
- **BDDステップ完了判定**:
  - 現在完了したファイル名（例: `01-api-foundation.feature`）の番号を確認せよ。
  - 同ディレクトリ（`server/features/{TicketName}/`）内に、より大きな番号の `.feature` ファイルが存在するか検索せよ。
- **分岐処理**:
  1. **後続ステップがある場合（一部完了）**:
     - `tickets/active/` 内の該当チケットファイル（`.md`）のDoD（Definition of Done）にチェックを入れる。
     - **注意**: ファイルの `mv`（移動）は絶対に行わず、`active` に留めよ。
  2. **最大番号のBDDが完了した場合（全工程完了）**:
     - `scripts/update-adr-index.sh` を実行し、ADR目次を最新化せよ。
     - `tickets/active/` 内の該当チケットを `tickets/done/` へ移動（`mv`）せよ。

## 確認事項（AIへの最終チェック）
1. [ ] **テスト数値**: ADRの末尾にカバレッジ率が明記されているか？
2. [ ] **シンボル地図**: `docs/current_symbols.md` に今回の新設パッケージ/関数が含まれているか？
3. [ ] **進捗整合性**: 後続ステップがある場合、チケットはまだ `active` に残っているか？
4. [ ] **次への布石**: 次に実装すべき `.feature` ファイル名を提示したか？

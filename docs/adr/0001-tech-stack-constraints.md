# ADR-0001: 技術スタックの採用と実装上の制約

## Status
Accepted

## Context
本プロジェクトは高負荷なゲームサーバーを対象としており、パフォーマンスの予測可能性とデバッグの容易性を最優先する。

## Decision
以下の技術スタックと実装ルールを厳守する。

### 1. 言語・フレームワーク (Go / Gin)
- **Gin**: 軽量性を維持するため、ミドルウェアの追加は最小限とする。
- **Error Handling**: `errors.Is` および `errors.As` を使用したラップを徹底する。独自の `panic` は禁止し、必ず `error` を返却する。

### 2. データベース (PostgreSQL / sqlx)
- **No ORM**: パフォーマンス最適化とクエリの透明性のため、Gorm等のフル機能ORMは使用せず、`sqlx` または標準の `database/sql` を使用する。
- **Raw SQL**: 複雑な結合や集計は、コード内で生のSQLとして記述し、AIに生成させる際は実行計画を意識させる。

### 3. キャッシュ戦略 (Redis / Memcached)
- **Redis**: ランキング、ギルドデータ、Pub/Subなど、永続性や複雑なデータ構造が必要なものに使用。
- **Memcached**: セッション、マスターデータ、計算済みの一時キャッシュなど、高速なKey-Value参照に特化。
- **使い分け**: AIは「すべてRedis」を提案しがちだが、揮発性データについてはMemcachedを優先検討すること。

## Consequences
- 開発者は手動で構造体へのマッピングを書く手間が増えるが、DB負荷の制御が容易になる。
- AIの提案が「ライブラリ任せ」から「標準機能を活かした堅牢な実装」にシフトする。

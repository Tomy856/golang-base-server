# ADR-0002: 仕様駆動(SDD)および振る舞い駆動開発(BDD)の運用

## Status
Accepted

## Context
仕様と実装の乖離を防ぎ、AIが「何を実装すべきか」を正しく理解するためのプロセスを定義する。

## Decision
実装フェーズにおいて、以下の順序をスキップすることを禁止する。

### 1. 仕様定義 (Gherkin Syntax)
- コード変更前に、必ず `features/` ディレクトリ内の `.feature` ファイルを確認・更新すること。
- AIへの指示: 「まず仕様をGherkin形式で整理してから実装案を提示してください」と依頼する。

### 2. 構造設計 (Mermaid Diagram)
- 複雑なロジックや状態遷移は、`docs/diagrams/` 内に Mermaid 形式でドキュメント化する。
- Obsidian での閲覧を前提とし、ライブエディタで崩れないシンプルな記述を維持する。

### 3. テストファースト (BDD)
- `GoConvey` または標準の `testing` パッケージを使用し、仕様書の振る舞いをそのままテストコードに落とし込む。
- テスト関数名は `Test[機能名]_[期待する結果]` とし、日本語での説明コメントを付与すること。

## Consequences
- AIが「勝手に仕様を推測して実装する」ことを防げる。
- ドキュメント（Gherkin/Mermaid）が常に最新の状態に保たれ、開発者間の認識齟齬がなくなる。
